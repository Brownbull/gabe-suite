Why it never landed — and the one beat that lands it

You've asked for this many times. It bounced off for a structural reason, not a political one. The fix is one new beat: /gabe-red, between plan and execute. Nothing is split. Nothing that works is deleted.

The diagnosis

Three loops killed every previous attempt

the structural reason — this is the one
A beat asserts one terminal state. TDD has two contradictory terminal states of the same measurement.

One command cannot end with "the tests fail" and "the tests pass." So inside a merged execute, red is swallowed by the act that produced it — the only observable at the boundary is green, which is exactly the self-graded evidence the suite already refuses ("the verifier of the evidence is not its author").

Merged execute yields theatre by construction. That's why 3. Implement precedes 4. Run verification — and why nobody ever had to refuse test-first. The ordinal did it. No wording inside execute could ever have fixed a missing boundary.

The economic loop

Every test signal is derived from files that already exist — commit reports "84 passed"; review grades by file adjacency; feature claims by path glob. So a promised test has no reader — and a promise with no reader is never written, which means no reader is ever justified.

The political loop

Every prior attempt proposed a test beat justified by its artifact — which the standing correction correctly deletes as ceremony. A test-first mutation survives only as a development act that authors no doc. (The /gabe-execute deletion is the precedent FOR this beat, not against it.)

Nothing owned identity

Tests are the only durable artifact the lifecycle creates with no command present at their birth. Decisions get D[N]. Deferred items get P[N]. Phases get ids. Cards get slugs. Tests get nothing — born inside 3. Implement, a step with no registry write.

⚑ The live orphan — your contract is already written

evidence-doctrine.md:21 has said this for months: "failing-then-passing test (fails on base, passes on fix)" — and :15 defines a proof: test type. The red-green contract already exists in the suite. PLAN.json's schema simply never modeled the proof type, and execute only ever wrote the runtime shape. It has no schema slot and no reader. This isn't a new idea — it's a branch that never grew.

The beat

/gabe-red — its deliverable is a commit that fails

/gabe-red [phase]

Deliverable: a commit whose declared cases fail by assertion. It authors no doc — it writes test code and produces a commit.

Red is not perishable — it is unaddressable. A commit gives it an address. That reframe is what makes this real: git worktree add /tmp/x <red-sha> && pytest -k C147 re-derives the failure, by a stranger, months later. Re-runnability is why it counts as evidence.

And it buys something free that nothing else in the suite can answer about a single test:

git log -S "C147" → first commit → did it carry a RED: trailer?
     → has this case ever been observed failing?

A test that has never been red is not known to test anything.

fails by assertionRED — real evidence.
fails by import/collectionNOT RED — non-evidence. A broken test isn't a failing test.
passes on unchanged codeTAUTOLOGY — halt. The case asserts nothing.

The subtlety that makes the tautology guard actually fire: the stub returns — it never raises. A raising stub makes every case fail, so every case looks red and the guard is dead. A returning stub lets the assertion execute against a real wrong value, so assert True gets caught. The stub is what enables the guard.

Test identity — the gap closes

The id lives inside the test's own text

Not a path. Not a registry file. Not scoped to a phase (a case outlives its phase, and PLAN.json resets to {"phases": []} on archive).

it('C147v2 · clamps a negative quantity to zero', ...)
def test_clamps_negative_quantity_C147v2():

In the NAME, not a comment — so it rides the existing junit pipeline with zero new plumbing: <testcase name="test_clamps_negative_quantity_C147v2"> carries identity into every report the suite already parses, and -k C147v2 selects it. This is what finally retires /gabe-feature's identity-by-path globs — identity-by-location is exactly what a rename breaks. The corpus IS the registry: grep it; max+1 allocates.

The earlier "no command can own the id" finding was half wrong. It's true of a path — not of an id.

Not everything is TDD-able

The refactor — the one everybody gets wrong

A refactor is not "not TDD-able." A refactor's test contract is that the existing cases stay green. So red doesn't skip — it declares guards, with no NEW and no BUMP:

CASES: — · GUARD: C091, C147, C203 (behavior unchanged; these must stay green)
RED: n/a (guard-only — no new claim)

Machine-checkable, correct, and not a fake red — it's the true statement of what the change promises. Anything genuinely un-testable self-skips with an enumerated code that lands in the phase's record. Red never fakes a red.

The bill

TDD is not free — stated plainly

developer cost

MVP (min_cases = 1): +10–25 min before any product code. Enterprise: +45–90 min. Scale: hours — correctly; the tier call bought it.

The dominant cost isn't writing the case — it's the corpus search (5–15 min on a mature corpus). That is precisely the work you asked for, and precisely the work that gets skipped when nobody makes you print it.

Net add is smaller than gross: execute already writes tests post-hoc at step 4. Red moves most of that time rather than adding it. Honest net: +10–15% phase time at MVP, +20–30% at Enterprise.

The real cost isn't minutes. Red asks the hardest judgment — "what behavior makes this change necessary?" — at the moment of least information. On an unsettled design you'll rewrite the cases. Expect 2–3 version bumps per case on early greenfield phases. That's TDD's known failure mode, not a flaw in this design.

⚠ The over-claim to name out loud

This does not close the human-witness gap. No case id, no red commit, and no fresh-context evaluator is a human walking the build — that evaluator is an agent. The release page will look like it closes it: diagrams, videos, proof shots, a stakeholder audience — and not one human witness in the chain. That over-claim is the most likely way this gets mistaken for finished.

The tripwire: if /gabe-red ever prints a summary a developer reads instead of a failure a developer must fix, it has become the ceremony the correction deletes — and it should go.
